[#reader-paths]
== 读者路径

包复用生态至少对应四类行动入口：引入依赖、发布包、设计工具平台、构造评价模型。行动不同，所需对象也不同。

=== 技术选型工程师

技术选型工程师需要判断一个候选包是否适合当前项目。这个判断不是单一偏好判断，而是功能相关性、许可证、平台兼容性、维护状态、依赖图、安全状态、构件完整性和组织策略的共同结果。

这类读者的主路径是包发现、xref:part-object-model[对象模型, rel=depends-on]、xref:part-lifecycle[生命周期, rel=depends-on]、xref:part-ecosystem-studies[相关生态章节, rel=depends-on] 和 xref:package-level-evaluation[包级评价, rel=depends-on]。该路径提供四个判断边界：

* 包能被搜索到，不等于包适合当前目标。
* 包能被安装，不等于包能长期维护。
* 下载量和 Star 是社会证据，不是安全证明。
* Lockfile 固化的是一次求解结果，不是许可证、安全和平台兼容性的替代物。

=== 包作者与库维护者

包作者需要把源码项目构造成生态可消费对象。这个动作不止发布文件，还包括声明身份、依赖、许可证、入口、平台约束、文档、版本、弃用策略和发布身份。

这类读者的主路径是 xref:part-object-model[对象模型, rel=depends-on]、xref:part-lifecycle[生命周期, rel=depends-on]、xref:part-security-governance[安全治理, rel=depends-on] 和 xref:part-ecosystem-studies[相关生态章节, rel=depends-on]。该路径说明 Manifest 中每个关键字段服务哪个消费者判断，文档和 API 投影如何影响发现，provenance 和 trusted publishing 如何改变发布身份的可观察性。

=== 工具、平台与 registry 设计者

工具和平台设计者需要判断一个包生态系统中哪些对象必须存在，哪些数据必须被索引，哪些信号可以进入排序，哪些风险不能被分数抵消。

这类读者的主路径是 xref:part-object-model[对象模型, rel=depends-on]、xref:part-evidence-measurement[证据与度量, rel=depends-on]、xref:part-security-governance[安全治理, rel=depends-on]、xref:part-evaluation-scale[评价尺度, rel=depends-on]、xref:appendix-source-map[资料来源图, rel=depends-on] 和 xref:appendix-structure-vocabulary[结构词表, rel=depends-on]。该路径区分 Registry、Index、Repository、Discovery Surface、Resolver、Lockfile、Artifact、Provenance 和 Attestation 的职责。

=== 研究者与模型构造者

研究者需要从多个语言生态中抽象共同结构，同时保留各生态的领域事实。JS/TS、Rust、Python、Go、JVM/Kotlin、.NET、C/C++ 和 Swift 面对不同约束；同一个维度在不同生态中的指标口径可能不同。

这类读者的主路径是 xref:part-evidence-measurement[证据与度量, rel=depends-on]、xref:part-ecosystem-studies[生态对照, rel=depends-on]、xref:part-evaluation-scale[评价尺度, rel=depends-on]、xref:appendix-variable-dictionary[变量字典, rel=depends-on]、xref:appendix-ecosystem-matrix[生态矩阵, rel=depends-on] 和 xref:appendix-source-map[资料来源图, rel=depends-on]。该路径说明哪些变量可跨生态比较，哪些变量只能生态内比较，官方事实源和第三方聚合面如何分层，评价函数如何避免成为无来源总分。
