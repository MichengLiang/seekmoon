== 读者路径

本书服务四类读者。每类读者面对的行动不同，阅读路径也不同。

=== 技术选型工程师

技术选型工程师需要判断一个候选包是否适合当前项目。这个判断不是单一偏好判断，而是功能相关性、许可证、平台兼容性、维护状态、依赖图、安全状态、构件完整性和组织策略的共同结果。

这类读者应先读第一部、第二部和第三部，获得包发现、包管理、对象模型和生命周期。随后按项目语言阅读第五部对应生态章节，再阅读第四部和第七部中的包级评价部分。

读完后，读者应能区分以下判断：

* 包能被搜索到，不等于包适合当前目标。
* 包能被安装，不等于包能长期维护。
* 下载量和 Star 是社会证据，不是安全证明。
* Lockfile 固化的是一次求解结果，不是许可证、安全和平台兼容性的替代物。

=== 包作者与库维护者

包作者需要把源码项目构造成生态可消费对象。这个动作不止发布文件，还包括声明身份、依赖、许可证、入口、平台约束、文档、版本、弃用策略和发布身份。

这类读者应重点阅读第二部、第三部、第六部和相关生态章节。读完后，读者应能说明 Manifest 中每个关键字段服务哪个消费者判断，文档和 API 投影如何影响发现，provenance 和 trusted publishing 如何改变发布身份的可观察性。

=== 工具、平台与 registry 设计者

工具和平台设计者需要判断一个包生态系统中哪些对象必须存在，哪些数据必须被索引，哪些信号可以进入排序，哪些风险不能被分数抵消。

这类读者应读全书，尤其是第二部、第四部、第六部、第七部和附录。读完后，读者应能区分 Registry、Index、Repository、Discovery Surface、Resolver、Lockfile、Artifact、Provenance 和 Attestation 的职责。

=== 研究者与模型构造者

研究者需要从多个语言生态中抽象共同结构，同时保留各生态的领域事实。JS/TS、Rust、Python、Go、JVM/Kotlin、.NET、C/C++ 和 Swift 面对不同约束；同一个维度在不同生态中的指标口径可能不同。

这类读者应重点阅读第四部、第五部、第七部和附录。读完后，读者应能说明哪些变量可跨生态比较，哪些变量只能生态内比较，官方事实源和第三方聚合面如何分层，评价函数如何避免成为无来源总分。
