# MoonBit subprocess 能力与 DX 调查报告

这次我需要先更正刚才那个 spike 给人的第一印象：我刚才用 C FFI 自己写 `PATH` 查找、`popen`、UTF-8 转换、shell quoting，确实磕磕绊绊，偶然复杂度也很高；但这并不等于 MoonBit 目前完全没有 subprocess 能力。进一步调查后，结论更细：MoonBit 已经有官方 `moonbitlang/async/process` 方向，而且在最新 registry 发布版里可以直接跑 host process；但是它还没有形成 `zx`、`plumbum`、`xshell`、`CliWrap` 那种“现象级别、强推荐、开箱即用、大家都知道该用它”的高层 subprocess DX。

所以一句话结论是：**MoonBit 现在不是“没有 subprocess”，而是“有官方中低层 async process API，有若干第三方包装和 PTY/兼容层，但还没有成熟到主流语言里那种标志性脚本编排库的生态位置”。**

## 一、我刚才的 FFI 方式为什么看起来很糟

刚才的实验为了证明 MoonBit native 可执行文件能不能调用 host `jq`，我选择了最底层、最可审计但也最笨的路径：MoonBit `extern "C"` + native stub + POSIX `popen`。这条路能证明“native binary 能调用 host subprocess”，但它不是一个舒适的程序员接口，也不应该被当作生产级 subprocess 方案。

它暴露了几个典型偶然复杂度：

1. MoonBit `Bytes` 不是 C 的 NUL 结尾字符串，所以我一开始把 `String.to_bytes()` 当 `char*` 用，导致 `jq`、`bash`、`sh` 全都误判为 missing。后来必须在 C 里用 `Moonbit_array_length(bytes)` 取真实长度，复制后补 `\0`。

2. C 返回的字节不能随便当 MoonBit `String`，否则会出现乱码。最后必须显式使用 `moonbitlang/core/encoding/utf8`，MoonBit 到 C 用 `@utf8.encode`，C 到 MoonBit 用 `@utf8.decode_lossy`。

3. shell quoting 是巨坑。即使只是 `jq -c <filter>` 和 JSON payload，也要考虑参数拆分、单引号转义、stderr 捕获、exit code、PATH 查找。如果做成通用 subprocess 库，还要处理 Windows command line escaping、cwd、env、stdin streaming、stdout/stderr 分离、timeout、cancellation、process group、signal、zombie、编码等问题。

因此，你说“好像还很狭窄”是准确的。刚才那个 FFI spike 的价值只在于证明 native FFI 边界和 host subprocess 是可行的；它不代表 MoonBit 目前最好的 subprocess DX，也不应该作为推荐写法。

## 二、官方其实已经有 `moonbitlang/async/process`

我进一步调查了 `moonbitlang/async`。Mooncakes registry 上当前最新是：

```text
moonbitlang/async 0.19.4
build_status: success
downloads: 286297
license: Apache-2.0
repository: https://github.com/moonbitlang/async
```

我用 `moon fetch moonbitlang/async@0.19.4` 拉了发布版源码，里面存在完整的 `src/process` 包。它的 README 标题就是：

```text
Process Management (`@moonbitlang/async/process`)
Asynchronous process spawning and management for MoonBit with support for pipes, environment variables, and I/O redirection.
```

它提供的 API 表面包括：

```text
collect_stdout(cmd, args, ...)
collect_stderr(cmd, args, ...)
collect_output(cmd, args, ...)
collect_output_merged(cmd, args, ...)
run(cmd, args, ...)
spawn(group, cmd, args, ...)
spawn_orphan(cmd, args, ...)
wait_pid(pid)
read_from_process()
write_to_process()
redirect_from_file(path)
redirect_to_file(path)
graceful_cancel(timeout, signal?)
hard_cancel()
Process::wait()
Process::try_wait()
Process::cancel()
```

这已经不是“裸 FFI”级别了。它覆盖了主流 subprocess API 应该有的很多核心东西：参数数组、环境变量、继承环境开关、cwd、stdin/stdout/stderr 重定向、stdout/stderr 分离捕获、合并输出捕获、pipe、结构化并发 task group、取消策略、orphan process。

我还做了一个临时项目实测，不只是读文档。流程是：

```bash
moon new async_process_probe
moon add moonbitlang/async@0.19.4
```

然后 main 里写类似：

```moonbit
async fn main {
  let (code, output) = @process.collect_output_merged(
    "jq",
    ["-c", ".name"],
    stdin=@process.redirect_from_file("sample.json"),
  )
  println(code)
  println(output.text())
}
```

`sample.json` 内容：

```json
{"name":"moon","n":1}
```

验证命令：

```bash
moon check --target native
moon build --target native
moon run cmd/main --target native
```

输出：

```text
Finished. moon: ran 25 tasks, now up to date
Finished. moon: ran 57 tasks, now up to date
0
"moon"
```

这说明，在当前机器和当前工具链下，只要明确加 `moonbitlang/async@0.19.4`，官方 process API 可以正常调用 host `jq`。这比我刚才自写 FFI 的方式合理得多。

## 三、为什么一开始没有直接用官方 process

这里也有一个真实 DX 问题：本机 `moon ide doc '@moonbitlang/async/process'` 返回的是：

```text
package "moonbitlang/async/process" // moonbitlang/async@0.19.1

pub let unimplemented : Unit
```

这跟 registry 最新的 `moonbitlang/async@0.19.4` 发布源码不一致。也就是说，工具链本地可见符号缓存或默认安装状态显示的是旧版本/未实现状态，而 registry 最新包已经有完整 process API。对普通用户来说，这会造成非常严重的“发现失败”：你用 `moon ide doc` 查不到，就会以为官方没有；你看旧文档，也可能看不到；只有去 registry 拉最新源码或显式 `moon add moonbitlang/async@0.19.4` 才能确认。

这就是我认为当前 MoonBit subprocess 体验还不成熟的第一个证据：**能力已经在，但发现路径和版本对齐不够顺滑。**

## 四、第三方生态里有什么

我按 SOP 搜了 Mooncakes registry，并核验了几个高相关候选。

### 1. `moonbitlang/async/process`

这是目前最应该优先看的官方中层 API。它像 Go `os/exec` 加 async/structured concurrency，而不是像 `zx` 那样的 shell DSL。它是严肃 subprocess 基础设施，不是脚本糖。

优点：

- 官方包；
- native build 成功；
- 有 README 和测试；
- 支持 stdout/stderr capture、stdin、cwd、env、pipes、取消、task group；
- 实测可以跑 jq。

缺点：

- 需要显式依赖 `moonbitlang/async`；
- API 是 async 风格，对简单脚本略重；
- 不是 fluent builder，也没有 `$` template literal；
- 错误模型偏 exit code/raise，不像高级库那样封装成 Result 对象加丰富上下文；
- 文档/发现路径目前有错位。

### 2. `trkbt10/subprocess@0.2.0`

这个包描述是 Node.js `child_process`-like subprocess management for MoonBit。README 里有：

```moonbit
@subprocess.exec("ls -la")
@subprocess.exec_file("git", args=["status"])
@subprocess.spawn(...)
```

它提供 `ExecResult { exit_code, stdout, stderr }`，也提供 managed process manager 和 IPC。它比官方 process API 更像 Node 的 child_process，也更接近用户心中的 “subprocess 包”。

但它基于 `moonbitlang/async/process`。如果用户本地看到的是旧的 `@process.unimplemented`，会误判它不可用。它 registry build_status 是 legacy，downloads 比官方 async 少很多，目前还不能说是事实标准。

### 3. `sennenki/process@0.1.0`

这个更像 Rust/Go 风格的 process command builder。它有 `Command::new("moon").arg("version").stdout(Piped).output()` 这种接口，发布源码里有 C glue、测试、README。它比较接近 Rust `std::process::Command`，但目前版本低、downloads 很少、build_status legacy，不能说是生态主流。

### 4. `moonbit-community/pty@0.2.1`

这是 PTY 方向：pseudo-terminal spawning。它适合交互式终端进程，比如 shell、REPL、TUI 测试、终端自动化。它建立在 `moonbitlang/async/process.spawn` 上，README 里写了跨平台策略：Unix 用 PTY/self-helper，Windows 用 ConPTY。它 build_status success，说明这个方向已经有人在认真推进。

但 PTY 不是普通 subprocess；它解决的是交互式终端，而不是替代 `zx`/`xshell`。

### 5. `mizchi/x@0.4.0`

这个包 downloads 很高，定位是 Node.js backend compatibility layer for `moonbitlang/async`。它提供 `mizchi/x/process`，native 下委托给 `moonbitlang/async/process`，JS 下映射到 Node `child_process`。这是另一个很重要信号：MoonBit 社区已经意识到 process/fs/http/socket 等 host capability 要跨 native 和 JS 后端统一。

但它 README/TODO 也显示 process advanced API migration 仍在推进。它像兼容层，不像高层 shell scripting DSL。

## 五、和 zx / plumbum / xshell / CliWrap 比，MoonBit 缺什么

你给的表很准确。成熟语言里的好 subprocess 体验大致分几层。

### 第一层：底层安全执行

Go `os/exec`、Rust `std::process::Command`、Python `subprocess.run`、.NET `ProcessStartInfo` 解决的是：不要拼 shell 字符串，参数数组直接传给系统，能控制 env/cwd/stdin/stdout/stderr/exit code。

MoonBit 的 `moonbitlang/async/process` 已经接近这一层，而且是 async 版本。它不是缺。

### 第二层：中层便利封装

Node `child_process`、Python `sh`、Rust `xshell`、.NET `CliWrap` 开始提供更舒服的 wrapper：结果对象、捕获输出、失败自动抛错、pipeline、命令组合、日志、超时、取消、streaming、typed errors。

MoonBit 在这一层有候选，但还没稳定出一个大家都知道的推荐答案。`trkbt10/subprocess` 和 `sennenki/process` 在做这件事，但还不是事实标准；官方 `@process.collect_*` 也覆盖一部分，但不是完整 DX 包。

### 第三层：脚本语言级 DSL

`zx` 的核心吸引力不是“能跑进程”，而是：

```js
await $`git status --short`
await $`jq ${filter} ${file}`
```

它用 JS template literal 做 shell-like DSL，同时处理 escaping、stdout、stderr、pipe、Promise、错误上下文。`plumbum` 则用 Python 运算符重载和对象模型，把命令当一等对象组合。`xshell` 则在 Rust xtask 里用 `cmd!(sh, "...")` 宏提供编译期-ish 的命令拼装体验。

MoonBit 目前没有这个级别的标志性库。我没有找到一个类似：

```moonbit
let sh = @xshell.Shell::new()
let out = sh.cmd("jq", ["-c", filter]).stdin(json).read()
```

或者：

```moonbit
@sh.$("jq -c {filter}").stdin(json).text()
```

更不用说像 `zx` 那样成为“大家写 MoonBit 脚本时默认就用它”的文化入口。

## 六、这是不是 MoonBit 目前的缺陷

我认为是缺陷，但要分层说。

如果问题是：“MoonBit 有没有 subprocess 能力？”答案不是缺陷，官方 async/process 已经有。

如果问题是：“MoonBit 有没有一眼能发现、文档完整、API 稳定、主流推荐、能让人愿意用 MoonBit 写自动化脚本的 subprocess DX？”答案是：目前明显不足。这是生态缺口，也是 DX 缺陷。

这个缺陷的表现包括：

1. 官方 docs 入口不够显眼。你如果只读一般 language/toolchain 文档，不一定知道 `moonbitlang/async/process` 已经存在。

2. `moon ide doc` 和最新 registry 版本表现不一致。我本地查到的是 `async/process@0.19.1` 只有 `unimplemented`，但 `0.19.4` 源码和实测已经能用。这会严重影响发现。

3. API 名字和用法偏系统编程，不偏脚本体验。`collect_output_merged`、`redirect_from_file`、`TaskGroup` 很严肃，但没有 `zx` 那种“打开就写”的感觉。

4. 缺少事实标准 wrapper。第三方有 `trkbt10/subprocess`、`sennenki/process`、`mizchi/x/process`，但还没有一个库成为 `xshell` 对 Rust xtask 那样的默认选择。

5. 跨平台细节还在发展。PTY 包 README 明确写 Windows argv escaping 还有限制；process/compat 包 TODO 也显示 advanced API migration 仍在进行。

所以我的判断是：**MoonBit subprocess 的基础能力不是空白，但高层 DX 是当前短板。**

## 七、作为一门语言，它需不需要 subprocess

需要，而且几乎早晚必须做好。

原因很直接：MoonBit 如果只定位为浏览器/wasm/纯计算语言，那 subprocess 可以不是核心；但 MoonBit 现在明显在走 native CLI、agent、工具链、服务端、构建脚本、自动化、系统 glue 的方向。只要进入这些场景，subprocess 就不是可有可无。

具体场景包括：

- CLI 工具调用 `git`、`jq`、`curl`、`rg`、`fd`、`ffmpeg`、`plantuml`、`dot`；
- 构建系统或 codegen 调用外部 compiler/formatter/linter；
- agent 工具执行 shell 命令并捕获 stdout/stderr；
- DevOps 脚本替代 Bash；
- 测试工具启动 server/client 子进程；
- 语言工具链自身调用 linker、formatter、runtime；
- TUI/PTY 自动化；
- 和既有 Unix 工具生态组合。

这些都是现代开发语言绕不开的 host capability。没有好 subprocess，语言就很难成为“写工具”的语言，只能停留在“写纯库”或“写被别人调用的模块”。

所以我认为它是早晚会做，而且正在做。官方 `moonbitlang/async/process` 就是证据；`moonbit-community/pty`、`mizchi/x/process`、`trkbt10/subprocess` 是生态证据。问题不是会不会做，而是会不会把它包装到足够高的 DX 层，让普通用户不用踩 FFI、Bytes、shell quoting、Windows escaping 这些坑。

## 八、MoonBit 应该怎么走

我觉得 MoonBit 最合理的发展路径是三层：

### 第一层：官方稳定 `moonbitlang/async/process`

这是根基。它应该继续做好：

- argv 数组，不默认 shell；
- stdout/stderr capture；
- stdin bytes/string；
- env/cwd；
- timeout/cancellation；
- signal/process group；
- Windows quoting；
- typed error；
- structured concurrency；
- docs 和 examples。

这一层像 Go `os/exec`、Rust `std::process`、.NET Process/CliWrap 的底座。

### 第二层：官方或社区事实标准 `subprocess` wrapper

这层应该给 CLI 作者用，不一定 async-first 复杂暴露。比如：

```moonbit
let result = @subprocess.exec_file("jq", ["-c", ".name"], stdin=json)
if result.success() { println(result.stdout) }
```

它应该有：

- `exec_file` 和 `exec_shell` 明确区分；
- 默认防注入；
- `check=true` 自动把非零 exit code 变成错误；
- stdout/stderr 分离；
- merged output；
- stdin string/bytes/file；
- cwd/env；
- timeout；
- good error message；
- command-not-found 特判；
- Windows 行为文档。

`trkbt10/subprocess` 已经有这个方向的影子。

### 第三层：MoonBit 版 `xshell` / `zx`

这层才是你说的“现象级别 DX”。它可能不适合叫 subprocess，可能叫 `moonsh`、`xshell.mbt`、`moonbit-community/shell`。它应该服务于构建脚本、agent 工具、自动化流水线。

理想体验类似：

```moonbit
let sh = @shell.Shell::new(verbose=true)
let out = sh.cmd("jq").arg("-c").arg(filter).stdin(json).read_text()
sh.cmd("git").args(["status", "--short"]).run()
```

如果 MoonBit 语言未来支持更强的插值/宏/模板能力，还可以更接近：

```moonbit
let out = sh.$("jq -c {filter}").stdin(json).text()
```

但即使没有宏，也可以用 builder API 做得足够好。

## 九、我对“目前是否缺陷”的最终判断

我的最终判断如下：

1. **底层能力：不算缺。** `moonbitlang/async/process@0.19.4` 已经能 native 调用 host 命令，我实测了 `jq`。

2. **发现与文档：有缺陷。** `moon ide doc` 显示旧的 `unimplemented`，普通用户很容易误判。官方 docs 入口也不够突出。

3. **高层 DX：明显缺。** 还没有像 `zx`、`plumbum`、`xshell`、`CliWrap` 那样被广泛认可的、优雅的 subprocess 编排库。

4. **生态成熟度：正在形成。** `trkbt10/subprocess`、`sennenki/process`、`moonbit-community/pty`、`mizchi/x/process` 都说明社区已经开始补这个方向，但还没有收敛。

5. **未来必要性：非常高。** 只要 MoonBit 想认真做 native CLI、agent 工具、构建脚本、DevOps 自动化，就必须把 subprocess 做好。这不是边缘功能，是 host language 的基础设施。

## 十、我会怎么建议你后续推进

如果你现在只是要在 seekmoon 里做工具实验，优先用官方：

```moonbit
moon add moonbitlang/async@0.19.4
```

然后用：

```moonbit
@process.collect_output_merged("jq", ["-c", ".name"], stdin=@process.redirect_from_file("sample.json"))
```

不要再走我刚才那种自写 C FFI，除非你要研究 FFI 边界。

如果你想推动 MoonBit 生态，可以做一个小而精的 `moonbit-community/subprocess` 或 `moonbit-community/xshell` 风格库，底层委托 `moonbitlang/async/process`，上层提供非常舒服的 API。这个库很有价值，因为它正好填补“官方能力存在，但脚本 DX 不够”的空白。

我认为一个第一版可以非常克制：

- `cmd(name).args([...]).stdin_text(...).read_text()`；
- `cmd(name).args([...]).run()`；
- `cmd(name).args([...]).check()`；
- `shell(script)` 明确标红是 shell；
- `Result { code, stdout, stderr }`；
- `CommandNotFound`、`NonZeroExit` typed error；
- examples 覆盖 `git`、`jq`、`rg`；
- native target 先做好，JS/WASM 以后再说。

这比直接挑战 `zx` 全能力更现实，也更适合当前 MoonBit 生态阶段。

最终一句话：**MoonBit 不缺“能不能 spawn”的技术路线，缺的是一个让用户不用想 FFI、不用想 shell quoting、不用翻源码、不用猜版本的标准 DX 层。这个缺口很真实，而且我认为早晚一定会被官方或社区补上。**
